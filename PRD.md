# Catalyst4Sci PRD v0.2

## 1. 背景

当前以 Codex、Claude Code 为代表的 agent 已经能够完成代码生成、工具调用、单步分析与局部实验执行，但它们在科研场景中仍然存在明显短板：

1. 缺少长期目标管理能力。
2. 缺少结构化研究记忆，无法稳定跨多轮任务持续推进。
3. 缺少“下一步做什么”的显式决策机制。
4. 缺少文献调研、假设生成、实验执行、结果分析之间的闭环。
5. 缺少适用于长时运行任务的恢复、审计、预算控制与人工接管机制。

这导致现有 agent 更像“强大的研究助手”而不是“能够持续自主工作的研究者”。

Catalyst4Sci 的目标是构建一个面向科研工作的自治研究运行时，使通用 agent 能够在 IDE 或 CLI 环境中围绕研究目标持续开展工作，而不是依赖用户在聊天框中不断输入“继续”“下一步”“再试一次”。

## 2. 产品愿景

Catalyst4Sci 是一个面向科研任务的自治研究编排系统。它通过研究状态管理、下一步决策机制、skill 调度、实验执行和证据沉淀，把通用 agent 升级为可持续运行的科研 agent。

系统应当具备以下核心能力：

1. 在给定研究目标后，自主拆解问题并持续推进。
2. 模拟人类研究者的基本工作节奏，完成调研、假设、实验、分析与决策闭环。
3. 在没有用户持续输入的情况下自动决定下一步。
4. 结合 skill 库调用不同工具和流程完成研究工作。
5. 保留完整研究轨迹、证据链和决策依据，支持审计与恢复。

## 3. 产品目标

### 3.1 核心目标

1. 构建一套适配 Codex、Claude Code 等 agent 的自治研究执行框架。
2. 让 agent 能够在单个研究任务中连续执行多轮研究循环。
3. 让“下一步动作”由状态与证据驱动，而不是由用户聊天驱动。
4. 通过结构化记忆支持长期任务恢复、对比与总结。
5. 为后续扩展到多领域科研任务提供统一运行时基础。

### 3.2 第一阶段目标

第一阶段聚焦于一个最小可闭环科研场景：

1. 以机器学习实验型研究为主要验证场景。
2. 支持目标定义、实验设计、实验执行、结果比较、自动下一步决策。
3. 支持在 CLI 中启动、暂停、恢复和查看任务状态。
4. 支持通过 agent 和 skill 执行代码修改、实验运行、结果分析等动作。
5. 采用不依赖向量模型的基础记忆系统，优先保证可解释、可恢复和可调试。

## 4. 非目标

第一阶段不做以下事项：

1. 不追求全学科通用科研能力。
2. 不追求完全替代人类研究判断。
3. 不支持无边界、无预算、无约束的自主联网与实验执行。
4. 不在 v1 中构建复杂的多 agent 群体协作系统。
5. 不在 v1 中实现面向论文投稿的完整自动写作与发表流程。
6. 不在 v1 中支持高风险现实世界实验控制。
7. 不在 v1 中引入向量数据库或 embedding 作为系统基础依赖。
8. 不在 v1 中构建重语义记忆基础设施。

## 5. 用户画像

### 5.1 核心用户

1. 希望把 Codex、Claude Code 等通用 agent 改造成科研 agent 的开发者。
2. 需要自动调参、自动实验与自动结果比较的机器学习研究者。
3. 需要长期运行研究任务而不希望反复手动驱动的高级用户。
4. 需要为科研工作流构建底层平台能力的工具链团队。

### 5.2 用户特征

1. 具备工程或研究背景。
2. 理解实验、指标、假设与验证流程。
3. 关注结果质量、可追踪性和自动化程度。
4. 能接受系统在关键节点触发人工审批。

## 6. 核心问题定义

Catalyst4Sci 需要解决的关键问题不是“如何让 agent 更聪明”，而是“如何让 agent 持续工作且不失控”。

具体包括：

1. 如何把研究任务表示成可持续推进的状态机。
2. 如何让系统在每一轮结束后自动决定下一步动作。
3. 如何将文献、实验、结论、失败经验和开放问题沉淀为结构化记忆。
4. 如何在 IDE 或 CLI 中调用 agent 并保持长时任务稳定运行。
5. 如何在预算、风险、失败和中断条件下安全恢复或停止。

## 7. 产品原则

### 7.1 KISS

1. 先做单任务、单工作区、单研究循环的最小闭环。
2. 先采用可解释规则而不是复杂黑盒规划器。
3. 先把运行时建稳，再扩展学科和功能。
4. 先用文件系统和 SQLite 实现基础记忆，再考虑向量化增强。

### 7.2 YAGNI

1. v1 不做过早的多 agent 协作框架。
2. v1 不为远期复杂科研场景预埋大量抽象。
3. v1 只做当前明确需要的状态、决策、执行与记忆模块。

### 7.3 DRY

1. 文献调研、实验执行、结果评估均通过统一 task/skill 协议驱动。
2. 不为不同 agent 单独设计一套运行时语义。
3. 下一步决策逻辑在系统层统一，而不是散落在各类 prompt 中。

### 7.4 SOLID

1. Agent Runner 只负责调用 agent，不负责研究决策。
2. Decision Engine 只负责选择动作，不负责具体执行。
3. Memory Store 只负责持久化与检索，不直接耦合具体 agent 实现。
4. Skill 通过稳定接口扩展，不要求核心模块为每种 skill 修改逻辑。

## 8. 产品总体方案

Catalyst4Sci 的核心是一个自治研究循环：

1. 读取研究状态。
2. 评估当前阶段和目标差距。
3. 生成下一步候选动作。
4. 选择最优动作。
5. 通过 agent 或 skill 执行动作。
6. 评估结果。
7. 更新状态、记忆和证据。
8. 判断是否继续、暂停、请求人工或终止。

系统不应依赖“用户再次发消息”来触发下一轮，而应依赖上述显式循环。

在第一版中，系统的记忆与上下文机制采用以下方案：

1. 使用文件系统保存原始材料、日志、报告和 checkpoint 快照。
2. 使用 SQLite 保存结构化状态索引、对象关系、决策记录和实验摘要。
3. 使用分层上下文加载和规则检索控制上下文规模。
4. 不依赖向量模型、embedding 或向量数据库作为前置条件。

## 9. 核心场景

### 9.1 场景一：ML 实验自治闭环

用户给出研究目标，例如：

“在数据集 X 上提升模型 Y 的验证指标，在预算 Z 内完成若干轮实验。”

系统需要：

1. 生成初始研究计划。
2. 识别当前可优化变量。
3. 设计或修改实验。
4. 运行实验并记录指标。
5. 评估实验是否改进。
6. 自动选择下一轮实验或结束任务。

### 9.2 场景二：文献驱动的实验规划

用户给出一个研究问题。

系统需要：

1. 检索相关文献。
2. 提取主要思路、方法、指标和争议点。
3. 生成假设。
4. 设计实验验证这些假设。
5. 把文献证据与实验结果关联起来。

### 9.3 场景三：长时无人值守运行

用户在 CLI 或 IDE 中启动任务后离开。

系统需要：

1. 在无新聊天输入的情况下继续工作。
2. 在中断后恢复。
3. 在风险条件触发时暂停并请求人工介入。
4. 在任务结束时给出结构化报告。

## 10. 功能需求

### 10.1 研究目标管理

系统必须支持：

1. 定义研究目标、范围和成功条件。
2. 记录预算约束，例如时间、实验次数、资源消耗。
3. 记录研究上下文，例如代码仓库、数据集、已有论文或初始假设。

建议数据结构：

```ts
interface ResearchGoal {
  id: string;
  title: string;
  description: string;
  success_metrics: string[];
  constraints: GoalConstraint[];
  workspace: string;
}
```

### 10.2 研究状态管理

系统必须把研究过程显式建模为状态，而不是依赖聊天历史。

建议包含：

1. 当前阶段。
2. 当前计划。
3. 当前假设集合。
4. 已完成实验与结论。
5. 开放问题列表。
6. 当前预算状态。
7. 当前运行状态。

建议阶段：

```ts
type ResearchPhase =
  | "problem_framing"
  | "literature_review"
  | "hypothesis_generation"
  | "experiment_design"
  | "execution"
  | "analysis"
  | "decision"
  | "reporting";
```

### 10.3 自动下一步决策

这是产品最关键能力。

系统必须在每一轮执行后，基于当前状态自动输出下一步决策，而不是等待用户输入。

决策流程必须包括：

1. 生成候选动作。
2. 评估候选动作的收益、成本、风险和重复度。
3. 选择一个动作执行。
4. 记录选择理由与备选方案。

候选动作示例：

```ts
type NextAction =
  | { type: "search_literature"; query: string }
  | { type: "read_paper"; paper_id: string }
  | { type: "propose_hypothesis"; topic: string }
  | { type: "design_experiment"; hypothesis_id: string }
  | { type: "run_experiment"; experiment_id: string }
  | { type: "tune_parameters"; run_id: string }
  | { type: "compare_results"; run_ids: string[] }
  | { type: "summarize_findings" }
  | { type: "request_human_input"; reason: string }
  | { type: "terminate"; reason: string };
```

### 10.4 Agent 调用运行时

系统必须支持在 IDE 或 CLI 环境中调用不同 agent 执行任务。

必须支持：

1. 指定工作区。
2. 注入上下文与约束。
3. 接收 agent 输出。
4. 捕获工具调用、日志与异常。
5. 支持超时、取消、恢复。

适配对象包括但不限于：

1. Codex CLI
2. Claude Code
3. 后续其他兼容 agent

建议接口：

```ts
interface AgentRunner {
  run(task: AgentTask, context: AgentContext): Promise<AgentRunResult>;
  resume(runId: string): Promise<AgentRunResult>;
  cancel(runId: string): Promise<void>;
}
```

### 10.5 Skill 调度

系统必须通过统一协议调用 skill，而不是把 skill 当作纯文本提示词。

每个 skill 应包含：

1. 输入结构定义。
2. 输出结构定义。
3. 执行模式。
4. 风险等级。
5. 副作用说明。

建议结构：

```ts
interface SkillDefinition {
  id: string;
  name: string;
  description: string;
  input_schema: object;
  output_schema: object;
  execution_mode: "agent" | "script" | "tool" | "hybrid";
  risk_level: "low" | "medium" | "high";
}
```

### 10.6 实验执行

系统必须支持实验型动作的标准化执行。

至少包含：

1. 定义实验配置。
2. 触发实验运行。
3. 采集实验日志。
4. 读取关键指标。
5. 判断实验是否成功、失败或无效。

应支持的实验元信息包括：

1. 参数配置
2. 代码版本
3. 数据版本
4. 运行资源
5. 输出指标
6. 产物路径

### 10.7 文献调研与证据管理

系统必须支持把文献调研转化为可检索、可引用、可用于决策的结构化证据。

至少包括：

1. 文献检索
2. 文献筛选
3. 核心观点摘要
4. 方法与指标提取
5. 引用信息记录
6. 与假设、实验和结论建立关联

### 10.8 记忆与持久化

系统必须支持长时运行任务的结构化持久化。

应记录：

1. 研究目标
2. 当前状态
3. 假设集合
4. 实验记录
5. 证据条目
6. 决策记录
7. checkpoint
8. 周期性总结

第一版记忆系统采用“文件系统 + SQLite”的双层设计：

1. 文件系统负责保存原始内容。
2. SQLite 负责保存结构化索引和对象关系。
3. 所有研究关键对象必须有稳定 ID，并可跨模块引用。
4. 所有关键决策必须可查询，而不是只存在日志文本中。

文件系统建议保存：

1. 原始文献笔记和摘要
2. 实验 stdout/stderr 日志
3. 实验结果 JSON
4. 周期性报告
5. checkpoint 状态快照

SQLite 建议保存：

1. 研究状态索引
2. 假设索引
3. 实验证据索引
4. 决策记录
5. checkpoint 元数据
6. 对象关联关系

第一版不要求支持语义向量检索，优先支持以下基于规则和元数据的检索：

1. 按研究任务检索最近实验
2. 按 hypothesis 检索关联 evidence
3. 按 phase 检索当前相关 papers
4. 按时间、类型、状态筛选决策和实验
5. 按对象关系追溯某项结论的证据来源

### 10.8.1 分层上下文加载

为了控制长时研究任务的上下文规模，系统必须支持分层上下文加载。

建议采用三层结构：

1. `L0` 当前轮必需上下文
   包括当前目标、当前阶段、最近决策、当前最佳结果、当前待执行动作。
2. `L1` 当前任务工作记忆
   包括近期实验、活跃假设、关键文献摘要、当前阶段主要证据。
3. `L2` 长期档案记忆
   包括完整实验历史、旧报告、已淘汰方案、历史文献集合。

系统必须支持：

1. 优先加载 `L0`
2. 在动作需要时按规则扩展到 `L1`
3. 仅在必要时从 `L2` 检索补充信息
4. 记录一次检索到底加载了哪些上下文项

### 10.8.2 检索可观测性

系统必须记录上下文检索过程，用于调试和审计。

至少包括：

1. 查询输入
2. 命中的候选项
3. 排序依据
4. 最终注入上下文的项
5. 被舍弃的项及原因

### 10.9 暂停、恢复与接管

系统必须支持：

1. 暂停运行中的研究任务。
2. 从最近 checkpoint 恢复。
3. 在高风险动作前请求人工审批。
4. 人工接管后再将控制权交还系统。

### 10.10 研究报告输出

系统应在阶段结束或任务终止时输出结构化报告。

至少包括：

1. 初始目标
2. 执行过的主要动作
3. 实验结果摘要
4. 关键证据与文献结论
5. 当前最佳方案
6. 未解决问题
7. 后续建议

## 11. 非功能需求

### 11.1 可恢复性

1. 每轮循环结束后必须创建 checkpoint。
2. 中断后必须可恢复到最近一次稳定状态。
3. 恢复时必须验证工作区与状态一致性。

### 11.2 可审计性

1. 每次决策必须记录理由。
2. 每次实验必须记录输入、输出和状态变化。
3. 证据必须可追溯到文献、实验或观察来源。

### 11.3 可控性

1. 必须支持预算上限控制。
2. 必须支持动作级风险分级。
3. 必须支持人工审批与停止。

### 11.4 可扩展性

1. 新增 agent 适配器时不应修改核心循环。
2. 新增 skill 时不应修改决策引擎主逻辑。
3. 不同科研场景应共享统一状态与任务协议。

### 11.5 可调试性

1. 记忆检索路径必须可观察。
2. 上下文加载决策必须可追踪。
3. 研究状态、决策和实验摘要必须能通过本地文件和 SQLite 独立检查。

### 11.5 稳定性

1. 单轮执行失败不应导致任务整体损坏。
2. 崩溃应有重试、回退或终止策略。
3. 资源泄漏、循环空转和重复动作应被检测。

## 12. 系统模块设计

### 12.1 Agent Runner

职责：

1. 调用 Codex、Claude Code 等 agent。
2. 注入上下文与执行约束。
3. 管理执行日志、超时和恢复。

边界：

1. 不负责选择研究策略。
2. 不负责持久化研究结论。

### 12.2 Research State Manager

职责：

1. 维护研究阶段与状态快照。
2. 提供状态读取与更新接口。
3. 支持 checkpoint 和恢复。

### 12.3 Next Action Selector

职责：

1. 生成候选动作。
2. 对候选动作排序。
3. 输出最终动作和决策理由。

### 12.4 Skill Orchestrator

职责：

1. 解析 skill 定义。
2. 调度 skill 执行。
3. 统一处理 skill 的输入输出与失败。

### 12.5 Memory & Evidence Store

职责：

1. 保存研究记忆。
2. 保存证据、实验与决策记录。
3. 提供检索和摘要构建能力。

第一版实现要求：

1. 采用文件系统保存原始内容。
2. 采用 SQLite 维护索引和对象关系。
3. 采用规则检索和分层上下文加载。
4. 不依赖 embedding 和向量检索。

### 12.6 Loop Engine

职责：

1. 驱动整个自治循环。
2. 连接状态、决策、执行、评估和持久化。
3. 判断是否继续、暂停或终止。

## 13. 核心数据对象

建议至少定义以下数据对象：

```ts
interface ResearchState {
  goal: ResearchGoal;
  phase: ResearchPhase;
  hypotheses: Hypothesis[];
  current_plan: ResearchPlan | null;
  experiment_queue: ExperimentSpec[];
  completed_runs: ExperimentRun[];
  evidence_items: Evidence[];
  decisions: Decision[];
  open_questions: OpenQuestion[];
  budget: BudgetState;
  status: "idle" | "running" | "blocked" | "waiting_human" | "finished" | "failed";
  checkpoint_id: string | null;
}

interface Hypothesis {
  id: string;
  statement: string;
  source_refs: string[];
  status: "active" | "supported" | "rejected" | "stalled";
  confidence: number;
}

interface Evidence {
  id: string;
  type: "paper" | "experiment" | "observation" | "derived";
  summary: string;
  source: string;
  relevance_score: number;
}

interface Decision {
  id: string;
  timestamp: string;
  selected_action: string;
  rationale: string;
  evidence_refs: string[];
  alternatives: string[];
}
```

## 14. 自治循环定义

建议主循环如下：

```text
load_state
-> validate_state
-> generate_candidate_actions
-> select_next_action
-> dispatch_to_agent_or_skill
-> collect_outputs
-> evaluate_result
-> update_memory_and_state
-> checkpoint
-> decide_continue_or_pause
-> next_iteration
```

单轮循环必须输出：

1. 本轮前状态摘要
2. 候选动作列表
3. 最终选择动作
4. 执行结果
5. 评估结果
6. 状态更新
7. 是否继续

## 15. 停止条件与人工审批

系统必须定义明确停止条件：

1. 已达到目标或成功指标。
2. 预算耗尽。
3. 连续多轮无有效增益。
4. 连续多次执行失败。
5. 当前没有高价值下一步动作。
6. 动作风险超过阈值。
7. 需要人类做研究方向选择。

高风险动作需要人工审批，例如：

1. 删除大范围代码或数据。
2. 修改关键系统配置。
3. 执行高成本外部资源操作。
4. 调用高风险生产环境接口。

## 16. IDE / CLI 集成

### 16.1 CLI 模式

CLI 是 v1 的优先形态。

建议命令：

```bash
catalyst research start --goal "..." --workspace "./project"
catalyst research status
catalyst research pause
catalyst research resume
catalyst research report
```

CLI 需要支持：

1. 启动任务
2. 实时查看状态
3. 暂停与恢复
4. 导出报告

### 16.2 IDE 模式

IDE 模式作为 v1.5 或 v2 迭代方向。

IDE 面板应支持：

1. 当前研究阶段展示
2. 当前计划与下一步动作展示
3. 最近实验结果对比
4. 人工审批入口
5. 暂停、恢复、终止

IDE 应作为控制面板，不应承担核心自治逻辑。

## 17. MVP 范围

### 17.1 MVP 必做

1. 单研究任务运行时
2. CLI 启动、暂停、恢复
3. 研究状态持久化
4. 自动 next-action 规则引擎
5. 单 agent 适配层
6. skill 基础协议
7. 实验记录与结果比较
8. checkpoint 与恢复
9. 任务结束报告
10. 基于文件系统 + SQLite 的基础记忆系统
11. 分层上下文加载
12. 检索过程可观测

### 17.2 MVP 暂不做

1. 多 agent 协作调度
2. 跨学科通用科研能力
3. 复杂权限系统
4. 图形化研究工作流设计器
5. 大规模分布式执行平台
6. 向量模型驱动的语义记忆系统
7. 向量数据库接入

## 18. 成功指标

### 18.1 产品指标

1. 用户只输入高层目标后，系统可以自动连续推进多轮任务。
2. 用户不需要频繁在聊天框里发送“继续”。
3. 系统可以中断后恢复并保持研究上下文一致。
4. 用户可以随时查看当前状态、历史决策和下一步计划。

### 18.2 任务执行指标

1. 自动完成调研到实验的最小闭环。
2. 在预算内完成多轮有效实验。
3. 系统能避免明显重复动作和低价值空转。
4. 系统输出的结论具备证据链和实验依据。

### 18.3 工程指标

1. 新增 agent 适配器不影响核心循环。
2. 新增 skill 不要求重构状态或决策模块。
3. checkpoint 恢复成功率达到可接受水平。

## 19. 风险

1. 自治循环失控，产生重复低质量动作。
2. 文献理解质量不足，导致错误假设累积。
3. 过度依赖 prompt，导致行为不稳定。
4. 状态模型设计不清，后续扩展困难。
5. skill 协议不统一，执行路径碎片化。
6. 没有明确的预算和停止机制，导致资源浪费。
7. IDE/CLI 集成与 agent 适配存在实现差异，增加维护成本。
8. 如果记忆系统只有文件而缺少结构化索引，后续会退化为日志堆和 JSON 堆。
9. 如果过早引入向量模型，v1 可能被额外依赖和复杂度拖慢。

## 20. 开放问题

以下问题需要在下一轮设计中确认：

1. Next Action Selector 在 v1 是纯规则驱动，还是允许模型参与评分。
2. SQLite 的表结构如何覆盖 hypothesis、evidence、decision 和 checkpoint 的关系。
3. 文献调研模块在 v1 是直接联网检索，还是依赖外部 skill。
4. checkpoint 是否需要包含工作区快照摘要。
5. 是否在 v1 支持 Git 作为实验回滚辅助机制。
6. 哪些动作必须强制人工审批。
7. CLI 与 IDE 的优先级如何排序。
8. 何时引入 embedding 检索作为增强层，而不是默认依赖。

## 21. 结论

Catalyst4Sci 的核心价值不在于“能调用很多工具”，而在于：

**把科研过程从聊天交互提升为一个显式、可持续、可恢复、可审计的自治循环。**

参考 autoresearch 项目可以得到两个重要启发：

1. agent 只有在任务边界清晰、指标明确、循环稳定时，才真正适合长期自主工作。
2. 真正关键的不是单次执行能力，而是“每轮结束后如何决定下一步”。

Catalyst4Sci 需要在此基础上进一步引入：

1. 结构化研究状态
2. 决策引擎
3. 长时记忆
4. skill 协议
5. CLI / IDE 集成运行时
6. 风险治理与恢复机制

这将使通用 agent 从“科研助手”升级为“自治研究者”。

第一版在记忆层的明确策略是：

1. 不采用向量模型作为基础设施前提。
2. 采用文件系统 + SQLite 的轻量可解释方案。
3. 借鉴分层上下文和可观测检索思想，但不引入重语义依赖。
4. 待对象模型和检索需求稳定后，再评估 OpenViking 或 embedding 后端接入。
